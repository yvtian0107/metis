## Context

Metis 当前使用 `.env` + `godotenv` 读取基础设施配置（DB、端口、密钥、OTel），通过 `seed` CLI 和 `create-admin` CLI 初始化数据。这是一个开发者友好但终端用户不友好的模式。本设计将其替换为 WordPress 风格的两阶段安装向导。

当前启动流程：`.env` → `godotenv.Load()` → OTel init → CLI check → IOC → DB → seed.Run()（每次启动） → Apps → Gin → Listen`

目标启动流程：`读 metis.yaml` → `连 DB` → `检查 app.installed` → `未安装 → 安装模式 / 已安装 → 正常模式`

## Goals / Non-Goals

**Goals:**

- 用户下载 Metis 二进制后零配置启动，浏览器打开即看到安装向导
- 安装向导引导完成数据库选择、管理员创建、系统初始化
- 安装完成后 `metis.yaml` 自动生成（等同 WordPress 的 `wp-config.php`）
- 所有应用级配置存入 DB `SystemConfig` 表，通过设置页面管理
- 保留 seed 的增量同步能力（版本升级时补充新 roles/menus/policies）

**Non-Goals:**

- 不支持从安装向导切换数据库类型（SQLite → PostgreSQL 或反向），初始选择即最终选择
- 不支持 metis.yaml 热重载（修改后需重启）
- 不实现 SystemConfig 内存缓存（保持当前每次查 DB 的模式，后续优化）
- 不做安装卸载/重装功能

## Decisions

### D1: 配置文件格式选择 — YAML

**选择**: `metis.yaml`，使用 `gopkg.in/yaml.v3`

**替代方案**:
- JSON: Go 标准库支持，但无注释能力，对人类可读性差
- TOML: 需要额外依赖，社区不如 YAML 广泛
- 环境变量/CLI flags: 无法被安装向导自动写入文件持久化

**理由**: yaml.v3 已在 go.sum 中作为间接依赖，支持注释（向导生成时可加说明），WordPress 的 `wp-config.php` 也带注释，YAML 是 Go 生态常用配置格式。

### D2: metis.yaml 内容范围 — 最小化

**选择**: 只存放 DB 连接 + 安全密钥（4 个字段）

```yaml
# Metis Configuration — auto-generated by installer
db_driver: sqlite
db_dsn: "metis.db?_pragma=foreign_keys(1)&_pragma=journal_mode(WAL)"
jwt_secret: "<随机生成的 64 字符 hex>"
license_key_secret: "<随机生成的 64 字符 hex>"
```

**理由**: 学 WordPress — `wp-config.php` 只放 DB 设置 + auth keys。`server_port`、`otel_*` 等应用级配置全部存 DB，通过设置页面管理。密钥必须在配置文件中因为它们在 DB 连接建立之前就需要使用（JWT 验证）。

### D3: 安装状态检测 — 两级检测

**选择**: 文件存在性 + DB 标记双重检测

```
metis.yaml 不存在 → 安装模式（用默认 SQLite 启动，显示完整向导）
metis.yaml 存在 + app.installed != true → 安装模式（已配置但未完成安装）
metis.yaml 存在 + app.installed == true → 正常模式
```

**替代方案**: 只检查 DB 标记（但 PostgreSQL 场景下没有配置文件无法连接 DB）

**理由**: SQLite 用户可以零配置启动（默认值即可），PostgreSQL 用户必须先配置连接信息。两级检测覆盖所有场景。

### D4: 安装模式路由隔离

**选择**: 安装模式下 Gin 只注册 install 相关路由 + 静态资源，不注册任何业务路由

```go
if !installed {
    // 只注册: GET /api/v1/install/status, POST /api/v1/install/check-db, POST /api/v1/install/execute
    // + SPA 静态资源
    // 不注册: 任何 /api/v1/* 业务路由、JWT middleware、Casbin 等
} else {
    // 正常注册所有路由
}
```

**替代方案**: 注册所有路由但用 middleware 拦截 → 复杂且容易遗漏

**理由**: 最安全，安装模式下零攻击面。WordPress 也是类似模式——未安装时 wp-admin 只显示安装页面。

### D5: 安装完成后 — 不重启，热切换

**选择**: 安装 API 执行完成后，在同一进程内完成全部初始化（IOC 注册、seed、路由注册），返回成功后前端跳转到登录页

**替代方案**: 安装后要求用户重启进程 → 体验差，不像 WordPress

**理由**: WordPress 安装完成后直接跳到登录页，不需要重启 Apache/Nginx。Metis 应该做到同样的体验。实现方式：安装 handler 持有对 Gin engine 的引用，安装完成后动态注册所有业务路由。

### D6: Seed 拆分策略

**选择**: 将 seed 拆分为两个函数

- `seed.Install(db, enforcer)`: 安装时调用，全量初始化（roles + menus + policies + default configs + auth providers）
- `seed.Sync(db, enforcer)`: 每次启动调用，增量同步（只添加新的 roles/menus/policies，不覆盖已修改的）

**理由**: 版本升级可能新增菜单项或 API 策略，需要启动时同步。但默认配置值不应在启动时覆盖（用户可能已修改）。Install 和 Sync 的区别就在于是否写入 default configs。

### D7: OTel 初始化时机调整

**选择**: OTel 初始化推迟到 DB 连接之后，从 SystemConfig 读取配置

当前顺序: `.env` → OTel init → DB init
新顺序: `metis.yaml` → DB init → 读 SystemConfig → OTel init

**理由**: OTel 配置（enabled、endpoint、service_name、sample_rate）迁入 DB 后，必须先连 DB 才能读取。这意味着 DB 连接本身的 trace 无法被捕获，但这是可接受的 trade-off（WordPress 也不会 trace 自己的 DB 初始化）。

### D8: 前端安装状态检测

**选择**: 前端启动时调用 `GET /api/v1/install/status`，根据返回决定路由

```typescript
// App.tsx 启动时
const { installed } = await api.get('/install/status')
if (!installed) {
  navigate('/install')
}
```

**理由**: 简单直接。前端不需要关心 metis.yaml 是否存在——后端统一判断并通过 API 暴露安装状态。

### D9: 安装向导 UI 步骤

**选择**: 根据 SQLite/PostgreSQL 选择动态调整步骤

SQLite 流程（3 步）:
```
① 站点信息（名称） → ② 管理员账号（用户名/密码/邮箱） → ③ 完成
```

PostgreSQL 流程（4 步）:
```
① 数据库配置（主机/端口/用户/密码/库名 + 测试连接） → ② 站点信息 → ③ 管理员账号 → ④ 完成
```

首页：数据库选择（SQLite ● / PostgreSQL ○）

**理由**: SQLite 是零配置的，不需要数据库配置步骤。减少不必要的步骤让 SQLite 用户体验更流畅。

### D10: 新增 `internal/config` 包

**选择**: 新建 `internal/config/config.go`，负责 metis.yaml 的读取和写入

```go
package config

type MetisConfig struct {
    DBDriver        string `yaml:"db_driver"`
    DBDSN           string `yaml:"db_dsn"`
    JWTSecret       string `yaml:"jwt_secret"`
    LicenseKeySecret string `yaml:"license_key_secret"`
}

func Load(path string) (*MetisConfig, error)      // 读取
func (c *MetisConfig) Save(path string) error      // 写入（安装时用）
func DefaultSQLiteConfig() *MetisConfig             // SQLite 默认配置
```

**理由**: 配置文件的读写逻辑集中在一个包中，main.go 和 install handler 都引用它。

## Risks / Trade-offs

**[安装后热切换复杂度]** → 安装完成后在同一进程内初始化所有服务需要仔细管理 IOC 容器状态。Mitigation: install handler 接收 IOC injector，安装完成后按 main.go 相同顺序注册 providers 和路由。

**[metis.yaml 被误删]** → 如果用户删除 metis.yaml，下次启动会进入安装模式但 DB 已有数据。Mitigation: 检测到 DB 有 `app.installed=true` 但无 metis.yaml 时，提示用户恢复配置文件而非重新安装。

**[OTel 无法追踪 DB 初始化]** → OTel 推迟到 DB 之后初始化，DB 连接过程本身无法被 trace。这是 DB 配置存入 DB 的固有 trade-off，影响极小。

**[密钥在配置文件中]** → JWT_SECRET 和 LICENSE_KEY_SECRET 以明文存储在 metis.yaml 中。Mitigation: 文件权限应设为 0600（安装向导生成时设置），与 WordPress 的 wp-config.php 安全模型一致。
