## Context

当前侧边栏实现位于 `web/src/components/layout/sidebar.tsx`，只支持一级 Icon Rail 加二级 Nav Panel。AI 管理的数据 seed 虽然使用树形 `Menu` 模型，但实际只建到了 `AI 管理 -> 页面菜单` 这一层，因此前端只能把 `Agent / 知识库 / 工具 / 供应商` 平铺展示。

这导致 `工具` 成为一个“资源集合页”，其下的 `内建工具 / MCP 服务 / 技能包` 只能通过 `web/src/apps/ai/pages/tools/index.tsx` 里的页内 Tab 访问。对于用户来说，这三者其实是并列的可管理资源，不应该继续隐藏在一个页面内部。与此同时，AI 管理未来还会继续增长模板、会话、模型等入口，需要一个稳定的导航骨架，而不是继续依赖平铺菜单和局部 Tab。

本次变更会跨越后端菜单 seed、前端 app registry、通用 sidebar 渲染和 AI 模块路由，因此需要在实现前先明确导航层级语义、菜单顺序和兼容策略。

## Goals / Non-Goals

**Goals:**
- 为 AI 管理定义稳定的三级侧边栏结构，采用“二级能力域 + 三级资源项”的组织方式。
- 固化 AI 管理菜单顺序，确保 `智能体` 置顶，其余按 `知识 -> 工具 -> 模型接入` 排列。
- 将 `工具` 页中的 `内建工具 / MCP 服务 / 技能包` 从 Tab 提升为独立三级导航项。
- 在不破坏现有权限模型的前提下，让菜单 seed、前端渲染和路由注册共同支持新的三级结构。
- 为未来新增 `模板 / 会话 / 模型` 等菜单预留稳定扩展位。

**Non-Goals:**
- 不重做 AI 各资源页面本身的业务交互和视觉设计。
- 不修改 Provider、Knowledge、MCP、Skill、Agent 的业务 API 或权限语义。
- 不把所有详情页内的 Tab 都外提为导航；仅调整真正属于独立资源入口的 `工具` 域。
- 不触碰 `openspec/specs/` 之外的其他业务模块导航结构，例如 ITSM 或系统管理的菜单信息架构。

## Decisions

### 1. AI 管理采用固定的二级能力域顺序

AI 管理 SHALL 使用以下二级顺序：`智能体`、`知识`、`工具`、`模型接入`。

原因：这比按底层技术依赖排序更符合用户操作心智。用户通常先进入 Agent，再为 Agent 绑定知识和工具，最后才回到 Provider 这类基础设施设置。把 `智能体` 置顶也能让 AI 管理的默认落点更加明确。

备选方案：
- 按技术栈顺序排列（`模型接入 -> 知识 -> 工具 -> 智能体`）。未采用，因为更像实现顺序，不像产品入口顺序。
- 继续保持当前平铺顺序。未采用，因为无法支撑后续扩展，也不能解决 `工具` 域被 Tab 隐藏的问题。

### 2. 三级导航只承载“独立资源入口”，不替代详情页内部视图

Sidebar 的三级项 SHALL 表达独立可导航、可授权、可扩展的资源入口，例如 `内建工具`、`MCP 服务`、`技能包`。知识库详情中的 `Sources / Graph / Logs`、Agent 详情中的 `Config / Sessions` 仍保留在页内 Tab。

原因：如果把所有 Tab 都提升为导航，信息架构会迅速膨胀，而且详情页内部视图并不适合进入全局菜单树。三级导航应该解决“资源域分层”问题，而不是取代页面内部布局。

备选方案：
- 将 AI 内所有 Tab 统一升级为 sidebar 菜单。未采用，因为会把内容视图和资源入口混为一谈。

### 3. 通用 sidebar 从“二级列表”升级为“二级分组 + 三级菜单项”渲染

现有 `menuGroups` 仅用于给二级项加标题，无法表达真正的导航层级。前端 app registry 需要新增能表达分组及其子项的导航元数据，sidebar 根据该元数据渲染：
- 一级：App Icon Rail
- 二级：能力域分组
- 三级：该组下具体菜单项

原因：AI 三级导航不应通过前端硬编码特殊判断实现，而应成为通用导航能力的一部分。这样未来其他 App 如需三级分组也可以复用同一机制。

备选方案：
- 只为 AI 写死一个专用 sidebar 逻辑。未采用，因为会破坏 App 模块注册的一致性。
- 完全依赖后端 menu tree 自递归渲染，不保留前端 app registry。未采用，因为当前前端还依赖 `registerApp()` 和模块侧的导航元数据控制显示分组与文案，不适合在本次变更里整体迁移。

### 4. `工具` 域拆分为独立路由，并保留旧入口重定向

`/ai/tools` SHALL 作为兼容入口重定向到默认三级菜单页；新增独立路由用于承载：
- `/ai/tools/builtin`
- `/ai/tools/mcp`
- `/ai/tools/skills`

原因：从 Tab 提升为三级导航后，URL 需要具备可直达、可刷新恢复、可分享的稳定语义。同时保留 `/ai/tools` 重定向可以避免历史入口失效。

备选方案：
- 继续只保留 `/ai/tools`，通过 query/hash 区分页签。未采用，因为这仍是 Tab 语义，不是正式导航语义。

### 5. 后端菜单 seed 也同步变为三级树，避免前后端语义脱节

`internal/app/ai/seed.go` SHALL 把当前 `AI 管理 -> 页面菜单` 的结构调整为 `AI 管理 -> 二级目录 -> 三级菜单`，并继续沿用现有按钮权限挂载在最终菜单节点下。

原因：数据库菜单树本身支持层级，前端最终显示的结构应该与持久化菜单树一致。否则前端只是“看起来像三级”，后台菜单管理和权限树仍是扁平的，后期维护会非常混乱。

备选方案：
- 只改前端显示，不改 seed。未采用，因为菜单管理和权限树会与真实导航脱节。

## Risks / Trade-offs

- [Risk] 通用 sidebar 升级后，现有其他 App 的高亮和分组逻辑可能被影响。 -> Mitigation: 保持无分组子项的 App 继续走现有两级渲染路径，AI 先作为首个三级分组消费者验证兼容性。
- [Risk] `/ai/tools` 路由拆分后，旧书签或手动输入路径的用户可能遇到空白页。 -> Mitigation: 保留 `/ai/tools` 到默认三级页面的显式重定向。
- [Risk] 菜单 seed 改为三级后，已有数据库中的 AI 菜单层级需要迁移或补齐。 -> Mitigation: seed 使用基于 permission 的 upsert/补建逻辑，并优先通过 permission 定位既有菜单节点而非依赖名称。
- [Risk] `智能体 / 知识 / 工具 / 模型接入` 的命名虽然利于扩展，但会增加初次实现的 registry 元数据复杂度。 -> Mitigation: 先只让 AI 使用新的分组元数据，其他 App 保持原模型，逐步演进。

## Migration Plan

1. 先扩展前端 app registry 和 sidebar，使其同时兼容旧的二级菜单和新的二级分组/三级菜单结构。
2. 在 AI 模块中引入新的导航元数据和拆分后的工具路由，保留 `/ai/tools` 重定向。
3. 调整 AI 菜单 seed，将 `工具` 域及其他二级分组建为正式树形菜单。
4. 验证已有权限、激活状态、默认落点和国际化文案在新结构下正常工作。
5. 如需回滚，可先保留新的 sidebar 通用能力，但将 AI module 的导航元数据与 seed 恢复为旧的平铺结构，同时保留旧的 `/ai/tools` 单页实现。

## Open Questions

- 二级下只有一个三级项时，UI 是否始终展开显示，还是在视觉上弱化二三级差异，目前建议始终展示，保持结构稳定。
- `Agent` 最终文案是统一为 `Agent` 还是中文化为 `智能体`，实现时需要结合现有 i18n 和产品文案统一确定。
- 后续若要引入 `模型` 独立列表页，是继续挂在 `模型接入` 下，还是与 `供应商` 并列显示，当前设计预留但不在本次落地。
