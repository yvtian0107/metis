## Context

当前系统已完成部分 Agentic 语义升级（如审批后直接进入决策中状态、直接调度 SmartEngine），但体验仍分裂在四条链路：
1. 刷新链路：前端主要依赖手动刷新与 60 秒轮询观察结果。
2. 解释链路：SmartEngine 有决策上下文，但用户侧缺少结构化“为什么”。
3. 恢复链路：失败恢复偏工程入口，缺少统一产品化动作面板。
4. 提单链路：服务台工具已具备草稿流程，但主动追问与缺失补全能力不足。

约束：
- 不做向后兼容，不保留旧状态语义映射。
- 保持 Agentic 架构，不回退为手写确定性执行器。
- SQLite 单连接模式下禁止事务内长耗时智能调用，所有决策调度在提交后异步执行。

## Goals / Non-Goals

**Goals:**
- 建立事件驱动主路径，用户动作后 1-3 秒内可见状态变化。
- 建立统一可解释契约，让每轮决策均可读、可追责、可审计。
- 建立统一失败恢复编排，让失败态具备标准化恢复动作与权限控制。
- 将提单入口升级为对话式追问流程，减少信息缺失导致的驳回和返工。
- 建立决策质量指标闭环，支持按服务/部门持续优化。

**Non-Goals:**
- 不引入 WebSocket/SSE 作为本次前提；先以事件驱动 + 快速重取实现体感升级。
- 不重写 AI Tool 业务能力本身，只重构其触发与展示契约。
- 不引入旧状态兼容层或双写字段。

## Decisions

### 1) 采用“命令写入 + 事件分发 + 读模型投影”统一交互主链路

- 决策：将审批、驳回、动作完成等写操作统一沉淀为领域事件（如 `ticket.activity.decided`、`ticket.action.finished`、`ticket.decision.failed`）。
- 理由：消除“刷新推动流程”的隐式耦合，保证用户动作与系统推进之间的可追踪因果关系。
- 替代方案：继续以轮询任务为主推进。拒绝原因：体感滞后且链路不可解释。

### 2) 决策解释成为强制输出契约

- 决策：SmartEngine 每轮决策必须输出解释快照（依据、触发、结论、下一步、人工介入点），并投影到 ticket detail/read model。
- 理由：解释不是日志附属信息，而是产品主能力。
- 替代方案：仅在失败时展示日志。拒绝原因：用户在成功路径仍无法理解系统行为。

### 3) 失败恢复统一为 Recovery Orchestration

- 决策：定义恢复动作模型（重试、转人工、撤回）与权限矩阵；所有恢复动作记录时间线、操作者与结果。
- 理由：将“排障动作”提升为“可治理产品行为”，支持审计和运营优化。
- 替代方案：保留分散按钮和后台任务重试。拒绝原因：语义分裂、难以管控。

### 4) 对话式提单采用“缺失字段检测 + 增量追问”策略

- 决策：基于协作规范与服务表单约束，按优先级追问缺失信息；满足最低可决策条件后再提交。
- 理由：降低驳回率，避免用户一次性填完复杂表单的认知负担。
- 替代方案：继续静态表单。拒绝原因：无法体现 Agentic 协作价值。

### 5) 决策质量观测采用统一指标字典

- 决策：定义核心指标与口径（通过率、驳回率、重试率、平均决策时长、恢复成功率），并绑定到服务/部门维度。
- 理由：没有稳定指标就无法判断模型和流程优化是否有效。
- 替代方案：临时查询日志。拒绝原因：不可持续、不可比较。

## Risks / Trade-offs

- [风险] 破坏性状态迁移导致历史筛选异常。
  - Mitigation：一次性迁移脚本覆盖 ticket/activity/assignment/timeline；上线前全量校验采样。
- [风险] 事件驱动导致重复投递或重复执行。
  - Mitigation：事件幂等键 + dispatcher 防重窗口 + timeline 去重断言。
- [风险] 解释文本质量不稳定，影响用户信任。
  - Mitigation：解释输出 schema 校验 + 最低字段完整性门槛 + 回退为“人工接管提示”。
- [风险] 对话追问增加交互轮次，影响提交速度。
  - Mitigation：设置最低必要字段阈值，达到阈值可先提交后补全。
- [风险] 指标口径变更引发跨版本不可比。
  - Mitigation：在 capability 中冻结口径定义，版本变更走显式迁移。

## Migration Plan

1. 冻结旧状态口径，发布一次性迁移脚本（无兼容层）。
2. 上线事件驱动 dispatcher 与 read model projector，切断“刷新推进流程”路径。
3. 上线决策说明卡 API 合同与前端展示。
4. 上线恢复编排入口与权限矩阵。
5. 上线对话式追问策略（先灰度高频服务，再全量）。
6. 上线质量面板并建立周度巡检基线。

回滚策略：仅允许整包回滚到前一版本快照，不支持局部回滚（避免状态语义分叉）。

## Open Questions

- 对话追问达到“最低可决策条件”时，是否允许带风险标签直接提交？
- 解释卡是否默认对所有角色可见，还是先向审批人/管理员开放？
- 恢复动作中的“撤回”是否允许跨状态执行，还是仅限特定决策阶段？
