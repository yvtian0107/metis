# DESIGN.md

Metis 前端设计沉淀文档。当前内容以【供应商管理】为准，记录已经落地并验证过的页面结构、视觉语言和交互规则。

## 设计定位

供应商管理不是营销页，也不是高装饰性的品牌展示区。

当前采用的设计方向是：

- 精致、克制、系统化
- 轻量卡片，不做厚重“业务卡”
- 信息密度适中，优先可读和可扫视
- 品牌识别只做点到为止，不抢整体系统主题
- 列表页和详情页属于同一套后台工作台语言

## 统一主题基线

### 安静视觉语言（当前基线）

全站工作区采用安静、轻量的视觉语言，以供应商详情页为参考基线。核心原则：**容器退到背景，内容成为主角。**

workspace-* CSS 类是唯一的视觉重量控制层，组件和页面借助继承自动获得统一风格：

- `workspace-surface`（Card 基底）：半透明背景 + 弱边框，无 blur、无渐变、无阴影
- `workspace-page-header`：纯结构布局（flex + padding），底线分隔（`border-b`），无容器装饰
- `workspace-table-card`（DataTableCard 基底）：同 workspace-surface，轻量容器
- `workspace-table-toolbar`（DataTableToolbar 基底）：纯结构，底线分隔
- `workspace-panel`（侧边栏）：轻背景 + 右边框，无 blur
- `workspace-shell-bg`（根背景）：纯色混合，无渐变层

视觉关键词：

- 半透明背景（opacity 0.4-0.6）
- 弱边框（opacity 0.5）
- 无 backdrop-blur
- 无 box-shadow
- 无 linear-gradient
- 间距和底线做结构分隔，不靠容器包裹
- 通用 `Card` 标题到内容的默认节奏使用紧凑间距（`gap-4`）；表单/详情配置卡片不额外叠加 `CardHeader pb-4`

### 不再使用

- 毛玻璃风格（backdrop-filter: blur）
- 白色渐变背景（linear-gradient white 74%→54%）
- 双层阴影（inset + drop shadow）
- color-mix 高白混合边框
- 厚重圆角容器包裹页面头部（rounded-[1.5rem] + 背景 + 阴影）
- 业务页继续维持默认黑白灰壳层
- `ghost` / `secondary` 按钮沿用与工作台表面脱节的旧灰阶样式
- 详情页和表单页使用裸标题行，缺少统一页面头层级

### 保留

- 认证页（auth-*）独立视觉体系不受影响
- 安装向导（install-*）独立视觉体系不受影响
- body 背景微妙渐变保留——容器透明后它提供纵深感
- 主按钮可以带品牌强调，普通操作优先使用轻表面按钮

## 后台工作台 CRUD 基础模式

对应基础工件：`web/src/components/workspace/primitives.tsx`、`web/src/components/ui/data-table.tsx`

CRUD 管理页指“页面头部 + 搜索/筛选工具栏 + 表格 + 新建/编辑 Sheet + 行操作”的后台配置页面，例如用户、角色、渠道、组织、节点、SLA、优先级等。它们必须使用同一套安静工作台视角，而不是各自维护局部样式。

### 页面结构

- 根容器使用 `workspace-page`
- 页头使用 `workspace-page-header`、`workspace-page-title`、`workspace-page-description`
- 主操作放在页头右侧，使用小尺寸主按钮
- 表格容器只使用 `DataTableCard`，不要在外层再套一层卡片
- 搜索、筛选和轻量数量提示放在 `DataTableToolbar` 内

### 基础工件

通用 CRUD 视觉能力必须从 `@/components/workspace/primitives` 引用：

- `WorkspaceSearchField`：表格工具栏里的安静搜索输入，带搜索图标和清空按钮
- `WorkspaceStatus` / `WorkspaceBooleanStatus`：弱边框状态点，不使用高亮实心 Badge 表示启停
- `WorkspaceIconAction`：行内普通操作，icon-only，必须有 Tooltip 和 sr-only 文案
- `WorkspaceAlertIconAction`：行内确认类操作，例如删除，默认配合 `AlertDialog`
- `WorkspaceFormSection`：Sheet 表单内轻分组，标题使用小号大写/弱色
- `WorkspaceColorSwatch`：颜色类配置的固定尺寸色块

### 表格行

- 第一列优先承载对象名称和二级信息，例如编码、描述、颜色、短标签
- 右侧列保持稳定宽度，时间、数量、状态和操作不挤压主对象信息
- 行高保持紧凑，使用弱 hover，不做位移动效和阴影
- 状态用 `WorkspaceStatus`，只有业务分类、类型、范围等非状态信息才使用轻量 `Badge`
- 行操作默认 icon-only，不写“编辑 / 删除”文字；危险操作只在 Tooltip 与确认弹窗里表达文字

### Sheet 表单

- 新建/编辑优先使用 `Sheet`，不使用 `Dialog`
- 新增或重构表单时必须用 `WorkspaceFormSection` 按身份信息、权限/范围、时间承诺、描述等语义分组
- 复杂旧表单在触达时迁移，不为了视觉统一一次性重排所有业务字段

### 不再使用

- CRUD 页裸 `h2 + Button` 头部
- 表格外独立搜索条，或搜索按钮作为主要视觉元素
- 行内文字按钮反复展示“编辑 / 删除 / 查看”
- 启用/停用状态使用实心高亮 Badge
- 为了显得丰富而添加统计卡片、营销式说明卡片或重容器

## AI 内置工具管理设计

对应实现：`web/src/apps/ai/pages/tools/builtin.tsx`、`web/src/apps/ai/pages/tools/components/builtin-tools-tab.tsx`

AI 内置工具管理是 Agent 能力边界和工具运行时的配置工作台，不是工具市场页。

- 页面必须使用 `workspace-page` / `workspace-page-header`，不得回到裸 `h2` 标题
- Toolkit 分组卡片保持轻表面、弱边框和紧凑说明，只表达工具集名称、可执行数量、启用数量和用途
- Toolkit 详情使用 `Sheet`，工具启停、绑定影响、参数 schema 和运行时配置在同一上下文里完成
- 工具状态使用 `WorkspaceStatus` 弱状态点；`需配置` 是可执行前的配置状态，不是错误弹窗
- 有自有运行时的 Tool 在详情 Sheet 内展示独立运行时配置区；保存运行时只影响该 Tool，不影响绑定它的 Agent
- `itsm.service_match` 的服务匹配运行时归属 AI Tools：Provider、模型、温度、最大 Token、超时必须在工具详情中配置，不得在 ITSM 引擎设置重复出现

## ITSM 服务目录设计

对应实现：`web/src/apps/itsm/pages/services/index.tsx`、`web/src/apps/itsm/components/catalog-nav-panel.tsx`、`web/src/apps/itsm/components/service-card.tsx`、`web/src/apps/itsm/pages/services/[id]/index.tsx`

### 定位

服务目录是后台工作台里的服务定义入口，不是营销式服务市场。

视觉上必须延续全局安静工作区语言：

- 页面标题使用 `workspace-page-header` / `workspace-page-title`
- 左侧目录使用窄导航宽度，字体继承 `workspace-nav-item` 和 `workspace-nav-section-title`
- 左侧目录不重复展示页面标题，不使用大号局部标题或统计头部
- 目录计数使用右侧轻量数字，不使用胶囊卡片抢视觉重量
- 服务卡片使用 `workspace-surface`，不使用彩色顶条、品牌色头像、悬浮阴影或位移动效
- 卡片只保留服务名称、编码、引擎、启停状态和更新时间

详情页与列表页属于同一套后台工作台语言。基础信息、参考路径/策略草图、健康检查等模块使用轻表面与弱分隔；动作和知识文档继续复用 DataTableCard，不在表格外再套重卡片。

服务详情页是高密度配置工作台，不是宽松表单页：

- 基础信息使用紧凑配置栅格，服务名称、服务编码、所属分类、SLA、状态必须在同一组内可扫视
- 引擎类型在服务创建后不可编辑，详情页只在页面标题区作为只读身份标签展示，不进入基础信息编辑表单
- 状态是基础信息行尾的紧凑二元开关单元，Switch 与状态文案在同一控件块内居中对齐
- `所属分类`、`SLA 模板` 这类短选择器使用明确宽度，不占据半屏列后只显示短控件
- 描述和协作规范可以保留较宽输入区，但垂直间距要压缩，按钮留在表单尾部同一行
- smart 服务的“参考路径/策略草图”和“发布健康检查”在桌面端左右布局：参考路径占主宽度，健康检查作为右侧窄栏；窄屏再堆叠
- 发布健康检查是生成参考路径时固化的发布诊断快照，详情页只展示已保存结果，不在每次打开页面时实时 loading
- 发布健康检查没有保存结果或 `items` 为空时显示中性“暂无检查结果”，不能默认显示“有风险”
- 动作和知识文档的主操作按钮放在标题行右侧，不在标题下方额外放空工具行
- 服务详情内表格使用轻表头、弱分隔线和紧凑空态；空态高度控制在局部表格内，不全局修改 DataTable 默认高度

## ITSM 服务台设计

对应实现：`web/src/apps/itsm/pages/service-desk/index.tsx`

### Chat Workspace 共用基线

AI 管理智能体对话与 ITSM 服务台对话共享同一套 Chat Workspace，不允许再在业务页面内各自维护完整的对话框、消息渲染、输入框、图片上传或侧边会话栏。

对应实现：`web/src/components/chat-workspace/`

共享边界固定为：

- `ChatWorkspace` 负责对话壳层、页头、滚动区、消息列表、错误区、跳到底部和输入区布局
- `ChatHeader` / `AgentIdentity` 负责顶部智能体身份、状态点和业务动作区；业务页面只传身份和动作，不重新写一套页头
- `SessionSidebar` 负责会话列表、新建、选中态、加载态、空态，以及 AI 管理需要的日期分组、重命名、删除和折叠
- `ChatComposer` 负责文本输入、IME 安全回车发送、Shift+Enter 换行、停止生成、图片粘贴/选择/预览/移除
- `MessageTimeline`、`ThinkingBlock`、`PlanProgress` 归属共享层；业务页面不得复制消息流、推理块、计划进度、工具活动或 Surface 样式
- 业务差异必须通过 `surfaces` 注册。ITSM 的 `itsm.draft_form` 是业务 Surface，不进入共享组件内部硬编码

图片能力是对话系统的基础能力，不是 AI 管理独占能力。AI 管理和 ITSM 服务台都必须通过共享 Composer 走同一套图片预览、上传和发送协议；上传失败时保留当前文本和图片预览，并给出用户可见错误。

ITSM 可以保留自己的产品特色，但必须挂在共享结构上：服务台欢迎舞台、服务受理岗身份、草稿确认 Surface 和工单提交反馈属于 ITSM 业务配置；页头、侧边栏、消息区、输入区和图片输入不另起风格。

Chat Workspace 精修后的核心布局不得靠业务页面裸 `className` 修补。业务页面只能选择共享语义：

- Composer `variant`：`default` 用于标准对话输入；`stage` 用于欢迎舞台大输入；`compact` 用于更密集的工作台对话
- Composer `maxWidth`：`narrow` / `standard` / `wide`，ITSM 欢迎态固定使用 `stage + standard + service-desk`
- Composer `attachmentTone`：`chat` 表示普通图片上下文；`service-desk` 表示截图、报错图、审批材料等服务诉求附件
- Workspace `density`：`comfortable` 用于 AI 管理普通对话；`workbench` 用于 ITSM 服务台这类业务处理对话
- Workspace `messageWidth`：默认 `standard`；只有需要展示更宽业务 Surface 时才使用 `wide`
- Workspace `composerPlacement`：对话中使用 `docked`；未来需要浮动输入时使用 `floating`
- Sidebar `variant`：AI 管理使用 `ai` 保留日期分组、重命名、删除、折叠；ITSM 使用 `service-desk`，选中态弱高亮，避免看起来像二级导航

消息流必须使用 Timeline 模型，而不是问答成对模型：用户消息、图片、AI 回复、工具调用、reasoning、plan progress、业务 Surface 按可见消息时间线渲染，未形成助手回复时也必须保留用户消息。空态只允许出现在没有任何可见消息且不 busy 的会话中。Streaming 状态要有轻量“正在处理”反馈，不能只依赖文本突然出现。错误、重试、继续生成必须可发现，但不能成为对话主视觉。

ITSM `itsm.draft_form` Surface 是“服务台智能体整理出的服务申请草稿”，不是普通后台表单卡片。四态规则固定为：loading 展示草稿整理中的骨架态；editable 展示草稿标题、摘要和可编辑确认表单；submit error 在卡片内部展示；submitted 突出工单编号和后续处理提示。

服务台是 ITSM 的 Agentic 主入口，不是新建工单页，也不是服务目录页。

信息架构固定为：

- 工作台：服务台、我的工单、历史工单
- 服务管理：工单监控、服务目录、优先级管理、SLA 管理
- 系统配置：智能岗位、引擎设置

工单入口按视角分层：`我的工单` 只承载当前用户作为申请人提交的工单；`历史工单` 承载当前用户已经处理过的人工节点；`工单监控` 是跨用户的管理员运营视角，用于观察所有工单并处理卡单，必须放在 `服务管理` 分组，不承担个人待办职责。

服务台页面采用“左侧会话栏 + 中央服务台舞台”的两栏结构：

- 左侧是纵向会话列表和新建入口，不使用横向会话条
- 中央只展示单一 IT 服务台智能体，使用状态点表达可用状态，不追加岗位状态文字
- 对话顶部必须明确显示“当前智能体：{智能体名称}”，避免用户只从图标或小字猜测服务角色
- 对话顶部不显示会话时间，也不提供点击式智能体下拉；未来多智能体入口采用横向智能体结构承载
- 首屏是浅色舞台式欢迎态：服务台头像、简短身份信息、大输入框和轻量建议词
- 对话能力复用 AI Chat 会话和 SSE 流式输出能力，进入页面不自动创建空会话
- 对话页容器复用 AI 管理聊天页的高度模型：根容器 `h-full`，消息区 `flex-1 min-h-0 overflow-y-auto`，输入区独立 `shrink-0`
- 输入框复用 AI 管理的 floating card 体感：textarea 自适应高度、底部工具行、发送按钮在右下角
- 会话选中态使用平面弱高亮，不使用浮层、阴影或强 3D 感
- 对话顶部不放常驻重试按钮，避免把系统操作放到服务台主视线里
- 未配置服务受理岗时必须显示明确未上岗状态，并把用户引向智能岗位
- Agentic/Smart 服务草稿确认必须走结构化 UI Surface：SSE 使用 `data-ui-surface` 原位更新 `itsm.draft_form`，先展示草稿整理态，再展示复用服务表单 schema 的可编辑确认表单；普通文本摘要不能承担主确认界面，也不能让用户感到“SSE 文本突然变成表单”。

服务台不得退回经典表单思路：不要在入口处先让用户选分类、选服务、填表；用户应先直接描述诉求，再由服务台智能体澄清、填槽、生成草稿，用户确认后沉淀为结构化工单与过程记录。

## ITSM 工单监控设计

对应实现：`web/src/apps/itsm/pages/tickets/index.tsx`

工单监控是服务管理下的管理员运营视角，用于 Review 全部 Agentic 与经典流程工单的运转健康，并快速处理卡单。

- 页面首屏必须先表达运营健康：活跃、卡单、SLA 风险、AI 异常、今日完成、Agentic 活跃、经典活跃
- 健康概括使用单条轻量指标带，不拆成大面积统计卡片，不做大屏化装饰
- 主表格按风险优先排序，第一列用弱状态点和细风险色条表达阻塞/风险/正常
- 表格必须同时展示工单、服务/引擎、责任方、当前步骤、等待时长、SLA 和更新时间，避免只做普通工单清单
- 筛选区保留关键词、风险分段、引擎、状态、优先级、服务；风险分段是当前页主操作，不藏进高级筛选
- 点击工单在右侧处置抽屉中处理，保持监控上下文不中断；复杂排查再进入详情页
- 处置抽屉只展示卡点原因、当前上下文、手工动作和最近轨迹，不复制详情页的完整证据结构
- 手工动作优先复用现有指派、取消、人工通过/驳回、智能工单覆写和重试 AI 能力
- 页面整体继续使用安静工作台语言：弱边框、轻表面、紧凑行高、无渐变大背景、无厚重阴影

## ITSM 智能岗位配置页设计

对应实现：`web/src/apps/itsm/pages/smart-staffing/index.tsx`

智能岗位配置页是 Agentic ITSM 的岗位编制工作台，不是松散的说明型设置页。页面只表达真正以 Agent 身份执行任务、绑定工具并产生操作轨迹的岗位；后端直接调用模型的子能力必须拆到独立菜单「引擎设置」。

- 页面标题必须使用 `workspace-page-header` / `workspace-page-title`，保存动作放在页头右侧
- 顶层标题使用“智能岗位”，不得再以“引擎配置”“智能工单引擎”作为页面主概念
- 岗位编制固定为“服务受理岗 / 流程决策岗 / SLA 保障岗”
- 岗位卡片状态贴近岗位名称展示，使用工作台弱状态点；不得使用右侧漂浮的强色图标胶囊，也不得让中文状态被挤压换行
- 岗位卡片按内容高度自然排列，不用流程决策岗的额外字段把其它卡片拉出大面积空白
- 岗位卡片区必须撑满工作区宽度，宽屏下三列在 `xl` 断点展开，不使用局部 `max-width` 造成右侧空白
- 服务受理岗绑定服务台智能体；服务匹配由 AI 管理的 `itsm.service_match` 内置 Tool 提供，不在 ITSM 引擎设置中配置
- 流程决策岗绑定流程决策智能体，岗位模式必须靠近流程决策岗
- SLA 保障岗绑定能读取 SLA 风险、规则并触发升级动作的智能体；SLA 超时事实仍由系统计时器判定
- 运行参数不得出现在智能岗位页；独立菜单「引擎设置」只承载 ITSM 领域内的参考路径生成、异常兜底与审计，未来经典引擎设置也放在同一个入口
- 引擎设置页不能用“Agentic 引擎”这类抽象引擎名做配置块标题；当前必须按用途拆成「参考路径生成」「异常兜底与审计」
- 引擎设置的用途卡片可以展示对应能力的健康状态，但状态必须贴近用途标题，使用工作台弱状态点，不使用右侧漂浮强色胶囊
- 引擎设置卡片使用左侧用途说明、右侧参数控件的配置带结构，不再使用标题区和表单区上下分离的大块卡片
- 「参考路径生成」使用单条紧凑配置带，Provider、模型、温度、重试和超时在宽屏同一行展示
- 「服务匹配」的 Provider、模型、温度、最大 Token 和超时归属 AI 管理的内置工具详情 Sheet；不得继承或暗用服务受理岗模型，也不得在 ITSM 引擎设置里重复出现
- 引擎设置的用途卡片必须统一撑满内容区，避免一张全宽、一张短卡造成右侧大块空白；卡片内部用稳定列宽控制输入控件，不靠卡片宽度拉伸控件
- Provider 和模型控件必须有明确宽度区间，不能因为外层卡片很宽而横向撑满
- 不再回到“受理引擎 / 决策引擎 / 路径引擎 / 运行保障”的旧引擎部件心智

## ITSM 经典流程编排页设计

对应实现：`web/src/apps/itsm/pages/services/[id]/workflow.tsx`、`web/src/apps/itsm/pages/services/[id]/workflow-preview.tsx`、`web/src/apps/itsm/components/workflow/`

经典引擎是流程设计工作台，不是传统详情页里的若干配置卡片，也不是 BPMN 图形控件。视觉方向是 Metis 主题下的精密控制台：浅色蓝灰、弱边框、细网格、紧凑配置密度、低噪声节点和上下文检查器。

- 经典服务详情首屏直接进入全高设计工作台，不再纵向堆叠基础信息、进件表单、工作流和动作卡片
- 工作台固定为左侧资源栏、中央横向画布、右侧属性检查器和画布内浮动工具条

- 右侧检查器承担全部上下文编辑：开始节点编辑进件表单，表单/处理节点编辑节点表单、参与人、输入输出，连线编辑结果、默认路径和条件
- 表单设计不再默认打开大 Sheet；使用嵌入式 Form Composer，以紧凑字段列表、字段属性分组和拖拽排序完成配置

- 主流程固定为从左到右，不提供旧纵向模式切换
- 空经典流程默认生成“开始 → 结束”骨架，不显示大面积空态

- 不再使用大面积彩色底、粗彩色边框、圆形/菱形混搭节点和重阴影表现流程类型

- 连线使用横向 Bezier 弱色线，处理完成/表单提交/默认路径/条件路径只用轻量标签表达
- 编辑态连线中点保留插入节点入口，优先让用户在上下文中扩展流程，而不是回到左侧重新拖拽
- 前端保存的连线默认路径字段必须使用后端真实字段 `data.default`，不得再写 `data.isDefault`
- 并行/包含网关必须在节点属性中配置 `data.gateway_direction`，取值只允许 `fork` 或 `join`
- 节点库不暴露当前无法完整配置或不可运行的半成品节点；等待场景统一使用 `wait` 节点承载定时/信号配置
- 服务详情预览和工单运行态查看必须复用同一套节点与连线视觉，不再各自维护旧样式
- 运行态用节点边框、状态图标、访问过的连线表达当前/完成/失败/取消/未到达；不依赖整卡透明度或强脉冲动画制造状态
- 右侧配置面板宽度保持 360-400px，按身份、执行、输入输出等语义分段；删除等危险动作固定在面板底部
- React Flow 样式只能使用 workflow 局部 class/CSS 变量覆盖，不影响 APM 拓扑或其它画布

## ITSM 工单管理处置详情设计

对应实现：`web/src/apps/itsm/pages/tickets/[id]/index.tsx`

### 定位

工单详情页首先是管理员处置台，不是信息堆叠型详情页。

首屏必须优先回答三个问题：

1. 这张单现在卡在哪个节点
2. 当前责任方是谁，风险或失败原因是什么
3. 当前用户能执行什么动作

### 结构规则

工单详情首屏采用“摘要 + 处置栏 + 证据区”的结构：

```text
处置摘要：工单诉求、当前节点、责任方、关键事实条
右侧处置栏：通过 / 驳回 / 指派 / 接管 / 重试 AI / 取消工单
主栏证据 Tabs：AI 依据 / 流程轨迹 / 变量 / 时间线
```

工单事实只保留能支撑管理员处置的字段：服务、优先级、申请人、来源、创建时间、SLA。流程、变量、时间线只作为证据与审计记录，不再首屏展开，也不承担主要解释职责。

处置摘要不写“当前没有明显风险，继续处理”这类泛化结论；没有真实判定依据时，宁可只展示工单诉求、下一步和关键事实。证据 Tabs 放在左侧主栏内部，不能被右侧处置栏高度拖出首屏。右侧处置栏按钮采用紧凑成组布局，避免一行一个大按钮造成空间浪费。信心度只作为右侧处置栏标题区的小 Badge，不单独占用一整块面板空间。当前可处理人工节点的主动作固定为“通过 / 驳回”，两者都必须要求处理意见，并把意见写入流程上下文供流程决策岗判断下一步。人工覆盖、指派、接管、重试等处置按钮保持同一尺寸。右侧处置栏不展示全局“完结/关单”动作；工单结束只能由流程推进或 smart 引擎 complete 决策产生。若未来需要强制完结，只能作为人工覆盖类高风险动作并要求理由，不得放入主按钮组。

smart 工单的 AI 信息必须融入管理员处置：

- 低置信状态转为可见的人工处理/接管任务，前置到右侧处置栏
- AI 置信度、下一步建议、推理摘要进入摘要区和 AI 依据 Tab
- 智能流程只在“流程轨迹”Tab 展示，不与主处置动作抢视觉重量

### 安全感规则

- AI 可以自动推进，但必须显式展示当前状态和下一步计划
- 低置信决策要前置接管入口，不把处置入口藏在时间线里
- 自动化动作要明确展示“执行中 / 已执行 / 失败”，失败时提供接管路径
- 管理员接管入口与普通处理动作并列呈现，但所有接管动作必须保留原因和审计记录
- smart 服务里的 `workflowJson` 在产品语义上是“参考路径/策略草图”，不是运行时严格执行的固定流程图

### 配置页规则

smart 服务配置页必须展示发布健康检查，但它是智能引擎运行前诊断，不是配置完整度评分。

健康检查只报告会导致智能引擎无法运行、决策歧义、引用失效或运行策略退化的问题，例如核心 Agent 缺失、协作规范为空、参考路径结构错误、参考路径动作失效、人工节点参与者明显不可解析。未配置自动化动作、未关联知识库、未配置兜底处理人、权限待确认等可选能力缺失，不能默认影响发布健康状态。

健康检查用于回答“智能引擎运行时是否存在明确歧义或风险”，不替代运行时审计。

## 认证页设计

对应实现：`web/src/pages/login/index.tsx`、`web/src/components/auth/auth-shell.tsx`

### 登录页定位

登录页不是营销落地页，也不是单纯的默认表单页。

当前采用的方向是：

- 控制台入口感，而不是“登录弹窗放大版”
- 品牌与系统语言并存，但品牌只做锚点，不做主视觉统治
- 桌面端采用双栏结构：左侧品牌叙事，右侧登录操作
- 移动端折叠成单列，优先保证输入与提交效率

避免的方向：

- 居中单卡 + 大面积发光 orb 的通用 SaaS 登录模板
- 过强玻璃拟态
- 为了显得先进而堆叠紫蓝科技特效
- 大段品牌介绍文案抢走登录动作焦点

### 布局规则

桌面端采用两栏：

1. 左栏：品牌识别、系统定位、一组轻量能力卡片
2. 右栏：登录表单、SSO、注册入口

结构如下：

```text
┌────────────────────────────────────────────────────────────┐
│ 品牌锚点 + 系统定位              登录标题 + 输入表单       │
│ 大标题 / 短说明                  用户名 / 密码 / 验证码    │
│ 3 个轻量能力卡                   登录 / SSO / 注册入口     │
│ 底部状态标签 / 版本信息                                   │
└────────────────────────────────────────────────────────────┘
```

要求：

- 左栏负责气质建立，不承担复杂交互
- 右栏必须保持明确的主任务路径
- 登录卡与外层舞台容器要有层级差，但不能做厚重卡片

### 视觉语言

- 背景允许有极弱结构线和轻量色团，但只能做氛围，不可喧宾夺主
- 外层容器是“薄雾面工作台”，不是高亮玻璃卡
- 登录卡使用细描边、低阴影、浅雾面背景
- 主色只用于焦点、按钮和品牌符号，不做大面积铺色

### 品牌表达

品牌表达保留在以下位置：

1. `AuthBrandLockup`
2. 左栏标题与短说明
3. 少量主色强调

不再使用：

- 满屏品牌色渐变
- 把品牌口号做成大段说明区
- 用背景特效替代品牌识别

### 表单规则

- 标题应直接说明动作，例如 `登录 Metis`
- 副标题只说明进入什么系统，不重复“请输入账号继续”这类空话
- 输入框使用主题 token，不单独维护 auth 专属灰阶体系
- SSO 按钮是次级动作，样式应回到系统按钮语言
- 错误提示必须内联、清晰、可即时感知

### 文案基线

- 优先使用明确动作：`登录`、`注册`、`验证`
- 少用模糊 CTA，如 `继续`
- 左栏说明控制在一段内，最多一句定位 + 一句补充
- 标签文案偏系统能力，不写营销化 slogan

## 供应商管理信息架构

采用两层结构：

1. 列表页：供应商网格卡片
2. 详情页：单个供应商的模型工作区

适用原因：

- 供应商本身需要快速总览
- 供应商下面还挂着模型管理，适合进入详情页做完整操作
- 列表页负责识别、状态、快速测试、进入详情
- 详情页负责模型的增删改、默认模型设置、同步

## 列表页设计

对应实现：`web/src/apps/ai/components/provider-card.tsx`

### 网格

供应商列表使用中等密度卡片网格。

原则：

- 卡片不能过小，否则品牌、状态、模型类型都会挤压
- 也不能过大，否则会喧宾夺主，破坏系统页面节奏
- 当前卡片高度为略高于菜单卡片的一档，用于容纳模型类型 chips 和底部状态区

### 卡片结构

当前卡片分为三层：

1. 头部
2. 模型类型摘要
3. 底部状态与快捷操作

结构如下：

```text
┌──────────────────────────────────────┐
│ [Logo] 名称  类型                ⋯  │
│        baseUrl                        │
│                                [chips]│
│ ──────────────────────────────────── │
│ ● 状态 · 相对时间          连通测试   │
└──────────────────────────────────────┘
```

### 头部规则

- 左侧使用真实供应商 logo，不再使用字母占位
- logo 放在中性容器里：`rounded-xl + border + muted 背景`
- 标题使用 `text-base font-semibold`
- 类型使用小号 `outline badge`
- `baseUrl` 是唯一保留的副信息
- 不显示 API Key

### 品牌表达

品牌表达只保留在两处：

1. 真实 logo
2. 类型 badge 文案

不再使用：

- 顶部粗色条
- 大面积品牌底色
- 品牌主导的卡片视觉

原因：

- 供应商是后台管理对象，不是品牌展示对象
- 真实 logo 已足够提供识别，不需要再用颜色重复强调

### 模型摘要规则

列表页中部不显示总模型数文案，例如：

- 不推荐：`模型数 14`

当前改为展示“各模型类型数量”，例如：

- `大语言模型 12`
- `向量模型 1`

这些信息使用轻量 `outline badge`：

- 小尺寸
- 中性色
- 不抢标题
- 右对齐排列

如果未来模型类型过多：

- 优先裁剪展示顺序靠前的类型
- 不建议把 chips 扩成第二层大模块

### 底部规则

底部使用 `border-t` 与上方内容分隔。

左侧：

- 状态圆点
- 状态文案
- 相对时间（如 `37m`）

右侧：

- 高频操作只保留一个：`连通测试`

中低频操作：

- 编辑、删除进入右上角 `⋯`

### hover 与操作区

- hover 只做轻微边框和阴影变化
- 不使用明显位移或厚阴影
- 卡片点击进入详情页
- 所有按钮和菜单必须挂 `data-action-zone`，避免点击穿透

### 引导卡

新增供应商使用虚线引导卡，样式比普通卡片更弱：

- `border-dashed`
- 中性背景
- `+` 图标 + 文案

引导卡必须与普通卡片使用同一尺寸体系，否则列表会断层。

## 智能体能力绑定

助手/编码智能体的能力绑定采用“集合优先”的交互模型：

- 外层摘要表示能力集合（Set），例如工具集合、MCP 集合、技能包集合、知识库集合、知识图谱集合
- 所有能力集合采用两段式旅程：管理集合时在单列列表勾选或取消集合，并默认启用集合内可用子项；配置时点击已选集合，再进入明细开关具体子项
- 外层已添加集合摘要采用最多四列卡片网格，展示集合名称、描述、已选数量/总数
- 点击集合摘要打开 Sheet，在 Sheet 中勾选或取消勾选该集合内的具体能力项
- 运行时只暴露已绑定集合中被显式勾选的可用子项；新增到集合中的子项不会自动授权给已有智能体
- 详情页按集合展示绑定摘要，不再只按资源类型展示 flat item 数量

视觉上延续工作台轻表面规则：集合摘要使用弱边框、轻背景、紧凑标题节奏；管理集合时优先用单列列表，不把待选集合或能力项直接全部平铺成拥挤网格。

## 组织与岗位管理设计

对应实现：`web/src/apps/org/pages/departments/index.tsx`、`web/src/apps/org/pages/departments/[id].tsx`、`web/src/apps/org/pages/positions/index.tsx`

### 定位

组织管理采用“组织主线”，岗位管理是组织上下文里的处理责任能力目录。

它首先服务 ITSM 流程参与人可达性：管理员应能清楚看到部门、岗位、成员三者如何绑定，而不是只看到孤立的 CRUD 表格。

### 组织架构页

组织架构页使用安静树形列表：

- 页面头只保留标题、视角标签、展开/收起和新增部门
- 部门编码内联在部门名称后，不再单独占一列
- 成员数使用小号文本，不做统计卡
- 状态使用低饱和圆点 + outline badge
- 行点击进入部门详情，展开按钮只控制树节点

### 部门详情页

部门详情页顶部是平铺身份区，不使用蓝色顶条或厚重概览卡。

头部信息聚合为：

- 返回入口
- 部门名称、编码、状态
- 负责人、上级部门、成员数、创建时间
- 编辑和更多操作

下方分区顺序固定为：可用岗位、部门成员、下级部门。

可用岗位展示岗位名称与当前成员占用；部门成员列表优先展示头像、用户名和邮箱，岗位用紧凑 chip 表达，主岗用星标区分。

### 岗位管理页

岗位页是岗位目录，不再只是名称/编码/状态的管理表。

列表行应展示：

- 岗位名称、编码、描述
- 启用该岗位的部门摘要
- 当前成员占用
- 状态与紧凑图标操作

岗位页不做大统计卡；覆盖范围和占用人数直接嵌入列表行，便于和组织详情互相印证。

## 详情页设计

对应实现：`web/src/apps/ai/pages/providers/[id].tsx`

### 头部信息区

详情页顶部不使用“概览卡”。

当前采用平铺信息区：

- 左侧：logo + 名称 + 类型
- 同一视觉层里展示 `baseUrl / 状态 / 协议 / 最后检测`
- 右侧：操作按钮组

原则：

- 不做单独的统计卡
- 不做大面积信息容器
- 头部只承担身份识别和全局操作

头部禁止：

- 返回按钮占一整块区域
- 重复页面标题
- 把状态、协议、最后检测拆成稀疏 description-list

这些信息应该压缩在同一行或同一信息层里，避免浪费垂直空间。

### 头部动作

当前保留三个操作：

- `连通测试`
- `同步模型`
- `编辑`

样式统一：

- `outline` 小按钮
- 同一高度
- 语义平级，不做主次冲突

当模块已经接入统一主题后，详情页头部外层优先复用 `workspace-page-header`：

- 返回路径放在头部信息区上方，作为轻量 breadcrumb/返回入口
- 标题、状态、类型标签放在同一信息块内
- 右侧动作区继续使用小尺寸按钮，但视觉要和工作台按钮语言一致

## 模型工作区设计

### 整体结构

详情页下半区不再用一个“模型管理”大面板包住所有内容。

当前结构是：

- 直接按模型类型拆分为多个分区面板
- 每种模型类型一个面板
- 桌面端一行 2 个面板

原因：

- 每种类型是天然分组
- 比单个大表格更适合扫视和定位
- 比 tabs 更适合比较不同类型是否有数据

### 分区面板风格

模型类型面板不是“强卡片”，而是“轻分区”。

规则：

- 弱边框
- 轻背景
- 大圆角，但不过度强调卡片感
- 标题区和内容区只用一条轻分隔线区分

不使用：

- 厚重标题条
- 大色块表头
- 数量统计 badge
- 很强的容器投影

### 分区头部

每个面板头部只保留三类元素：

1. 类型标题
2. 面板内搜索
3. 新建模型按钮

#### 搜索

搜索必须在各个类型面板内部，不放全局统一搜索。

原因：

- 用户搜索时往往针对某个类型
- 全局搜索会抢占页面结构中心
- 面板内搜索更符合“局部工作区”语义

当前搜索样式要求：

- 扁平
- 透明背景
- 无胶囊感
- 无明显边框盒子
- 仅保留轻底线

不允许使用高凸起、圆润胶囊、厚阴影输入框。

#### 新建模型

“新建模型”放在对应类型面板内，而不是全局工具栏。

点击后：

- 直接创建该类型模型
- `ModelSheet` 预填 `defaultType`

这是更短的操作路径，也减少了用户在表单里重复选择类型。

### 面板内容

每个面板内部仍然用表格展示模型，但表格只是承载数据，不应成为视觉主角。

当前列包含：

- 显示名称
- 模型 ID
- 状态
- 默认
- 操作

原则：

- 行高适中
- 默认列使用星标图标，不写冗余文案
- 操作列图标化
- 操作列标题和内容居中对齐

### 操作列

当前操作列统一使用图标按钮：

- 设为默认：`Star`
- 编辑：`Pencil`
- 删除：`Trash2`

原因：

- 文本按钮在窄面板里过于拥挤
- 图标化后更适合双列面板布局

规范：

- 操作图标水平居中排列
- 删除图标使用 `destructive`
- 破坏性操作必须二次确认

如果后续可用性不足，可补 tooltip，但不退回文字按钮。

### 空状态

每个类型面板都允许为空。

空状态设计：

- 不收起面板
- 保持固定高度
- 中间展示轻量空态

原因：

- 保持整体网格稳定
- 用户能明确知道该类型“目前没有模型”，而不是“不存在这个类型区域”

### 分页

每个类型面板独立分页：

- 每页 5 条
- 面板固定高度
- 底部有分页栏

分页栏规则：

- 可点击页码
- 保留上一页 / 下一页按钮
- 页数较多时用省略号

不推荐只显示 `3 / 3` 这种不可点击信息，因为跳页效率太低。

## Logo 规则

供应商 logo 当前使用 LobeHub 的静态 SVG 资源：

- 依赖：`@lobehub/icons-static-svg`
- 本地封装：`web/src/apps/ai/components/provider-logo.tsx`

选择静态 SVG 而不是重 React 组件包的原因：

- 依赖更轻
- 不引入额外 UI 体系
- 足够满足品牌识别

优先支持：

- OpenAI
- Anthropic
- Ollama

未知类型 fallback 才允许退回到文本缩写。

## 当前供应商管理设计关键词

如果后续继续扩展这个模块，必须保持以下关键词一致：

- 中性系统感
- 精致而克制
- 品牌识别点到为止
- 信息聚合但不做大统计
- 局部工作区优先于全局工具栏
- 表格可用，但不要让表格统治页面
- 卡片是轻容器，不是视觉主角

## 不再使用的旧模式

以下模式已经被当前供应商管理 UI 淘汰，不应继续沿用：

- 顶部品牌色条作为主视觉
- Avatar 首字母模拟品牌识别
- 详情页顶部“返回 + 标题 + 大概览卡”结构
- 全局统一模型搜索框
- 单个大面板包裹所有模型管理内容
- `模型数 14` 这种总数型摘要
- 强卡片感的模型类型面板
